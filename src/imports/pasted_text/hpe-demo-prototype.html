# Build an interactive Figma-style clickable demo prototype ## What to produce A single self-contained `hpe_demo_prototype.html` file in the current directory. When opened in a browser, it shows a **clickable prototype** of an RFID asset tracking and condition monitoring product. The user navigates by clicking hotspots on the screens — just like a Figma prototype. This is for a sales demo with a prospect called HPE (Hydropower Equipment, a South African mining equipment supplier). I'll click through it live with them in the room, then record a walkthrough afterwards. ## Hard requirements * **Single HTML file**. No build step, no npm, no frameworks. Vanilla HTML + CSS + JS. Tailwind via CDN is OK. Lucide icons via CDN is OK. * **Opens directly** by double-clicking the file. No server needed. * **Designed for 1440×900** viewport but responsive down to 1280×800. * **Click navigation only**, no auto-advance. The user controls the pace. * **Hotspots are visible on hover** — a subtle blue outline appears around clickable regions so I can see what's interactive (toggle this on/off with the `H` key). * **Browser-back support** — clicking the back arrow returns to the previous screen via the URL hash (`#screen-1`, `#screen-2`, etc). * **Persistent top bar** with: prototype name, current screen indicator (e.g., "3 of 9"), restart button, prev/next arrows. * **Smooth screen transitions** — 200ms cross-fade between screens. ## Visual style * Clean, flat, modern SaaS aesthetic. Think Linear, Vercel dashboard, Stripe. * Light mode. White surfaces, subtle borders (`#E5E7EB`), generous whitespace. * Primary accent `#1F4E78` (deep blue). Success `#10B981`. Warning `#F59E0B`. Danger `#EF4444`. * Sans-serif system stack: `-apple-system, "Segoe UI", Roboto, sans-serif`. * No emojis. Use Lucide icons via CDN. * Realistic-looking data everywhere. No "Lorem ipsum", no placeholder names. Use real SA mine names (Mponeng, Tau Tona, Driefontein, Kusasalethu, Kloof, South Deep) and asset IDs like DRL-04217, PMP-00184. ## Context for the product being demoed The product is an RFID asset tracking + condition monitoring system for mining equipment. The buyer (HPE) wants to track 7,000–10,000 hydro-powered rock drills and thrust legs through 7 movement events (Dispatched HQ → In transit → Received at mine → Dispatched UG → Returned from UG → Dispatched to HQ → Received at HQ), with data flowing into their Microsoft SQL Server. Phase 2 adds vibration + temperature sensors that must distinguish adjacent pumps shaking the same frame. Tags fit a 20mm × 80mm cavity in plastic drill handles, IP68, 500 g shock, 120 °C, 3–5 year battery. ## The 9 screens Each screen is a full-viewport "page". ` and render screens by toggling a `data-screen` attribute. Each screen is a hidden `
` that becomes visible when its number matches the current screen. * Hash routing: `window.location.hash = '#screen-N'` on navigation. Listen to `hashchange` to update. * Hotspots: any clickable region gets `class="hotspot"` and a `data-target="N"` attribute. Add a global click listener that intercepts these and routes via hash. When `H` is pressed (or a toggle button in the control bar is clicked), add a `.show-hotspots` class to the body that adds a `outline: 2px dashed #3B82F6; outline-offset: 4px` style to `.hotspot` elements. * Use real Lucide icons via `https://unpkg.com/lucide@latest`. Initialize with `lucide.createIcons()` after each screen change. * Use Chart.js via CDN for the line and bar charts (`https://cdn.jsdelivr.net/npm/chart.js`). Re-render charts when their screen becomes active. * For the map (Screen 3) and pump diagram (Screen 7), hand-write inline SVG — no external map libraries. * Keep the whole file under ~2500 lines. If you find yourself going over, simplify ornamental details, not navigation logic. * At the very top of the file, include a short comment block: filename, purpose, key keyboard shortcuts (`H` toggle hotspots, `←`/`→` prev/next screen, `R` restart, `1`–`9` jump to screen). ## Quality bar * Every screen must look like a real product, not a wireframe. * Numbers must be consistent across screens (e.g., DRL-04217's RMS is 4.2 g everywhere it appears). * No console errors when navigating between any two screens in any order. * The flow must be demonstrable in 3–5 minutes by clicking: Login → Dashboard → Map → click Tau Tona → Asset detail → click vibration chart → Pump physics → back → Acknowledge alarm → Alarms list → Reports → Work order success.
